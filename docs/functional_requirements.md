# Функциональные требования (Functional Requirements)

В данном документе описаны основные сценарии использования (Use Cases) и функции системы Secure-IoT Warehouse.

## 1. Участники (Actors)
*   **Датчик (IoT Device):** Устройство, собирающее телеметрию (температура, геркон).
*   **Оператор склада:** Пользователь, осуществляющий мониторинг через интерфейс.
*   **Администратор ИБ:** Сотрудник, ответственный за мониторинг сетевой безопасности и доступ к устройствам.
*   **WMS (Система управления складом):** Внешняя система-потребитель данных.

## 2. Сценарии использования (Use Cases)

### UC-1: Регистрация инцидента сетевой безопасности (Focus: Security/Networks)
**Акторы:** IoT-шлюз, Администратор ИБ.
**Предусловие:** Датчик пытается подключиться к MQTT-брокеру.
**Основной поток:**
1. Датчик инициирует соединение.
2. Система проверяет наличие TLS-сертификата устройства.
3. Система обнаруживает попытку подключения с невалидным сертификатом (или попытку сканирования портов).
4. Система блокирует IP-адрес источника на уровне сетевого экрана.
5. Система формирует инцидент и отправляет уведомление Администратору ИБ.

### UC-2: Мониторинг параметров микроклимата
**Акторы:** Датчик, Оператор склада.
**Основной поток:**
1. Датчик публикует данные (температура/влажность) в топик MQTT каждые 60 секунд.
2. Backend-сервис проверяет данные на соответствие заданным пороговым значениям.
3. При выходе за границы (например, > +25°C) система меняет статус зоны на «Тревога».
4. Данные сохраняются в БД для последующей предиктивной аналитики.

### UC-3: Синхронизация данных с WMS
**Акторы:** Backend-сервис, WMS.
**Основной поток:**
1. Система фиксирует срабатывание датчика открытия ворот.
2. Система формирует запрос через REST API к внешней WMS.
3. WMS сопоставляет время открытия с плановым графиком отгрузок.
4. В случае несовпадения WMS возвращает статус «Несанкционированный доступ».

## 3. Функциональные требования (FR)


| ID | Описание | Приоритет |
| :--- | :--- | :--- |
| **FR-01** | Система должна поддерживать протокол MQTT версии 5.0 для связи с устройствами. | High |
| **FR-02** | Система должна обеспечивать шифрование трафика по протоколу TLS 1.3. | Critical |
| **FR-03** | Система должна поддерживать ролевую модель доступа (RBAC) для управления складскими зонами. | Medium |
| **FR-04** | Система должна хранить историю логов подключений устройств не менее 3-х месяцев (требование ФСТЭК). | High |

## 4. Нефункциональные требования (NFR)
*   **Производительность:** Время обработки входящего сообщения от датчика не должно превышать 200 мс.
*   **Масштабируемость:** Поддержка одновременного подключения до 10 000 IoT-устройств.
*   **Надежность:** Дублирование MQTT-брокеров для исключения единой точки отказа.
